# 05 - 开源项目开发指南

> 返回 [📖 文档索引](README.md)

本篇教你如何用 Claude Code（CC）/ OpenCode 上手一个**陌生的开源项目**，并一步步走到提交 Pull Request（PR，向原项目提交改动的请求）。读者假定是完全的初学者，每一步都给出可直接照做的示例 prompt 或命令。

前置阅读：[01-安装与环境准备.md](01-安装与环境准备.md)（装好工具）、[04-常用Skill与扩展.md](04-常用Skill与扩展.md)（了解 `/init`、`/code-review` 等）。

---

## 一、整体心法

用 AI 参与开源开发，记住这五条原则，能少踩很多坑：

1. **先理解、再改动**：让 AI 先把项目读懂、把方案说清楚，再动手写代码。
2. **小步前进**：一次只做一件小事，改一点、验证一点，而不是一口气改一大片。
3. **每步可验证**：每次改动后都能跑测试 / 跑应用确认它真的生效（见 `verify`、`/code-review`）。
4. **敏感操作先确认**：删文件、改配置、提交、推送这类操作，先看清楚 AI 要做什么再放行。
5. **不泄露密钥**：不要把 API key、密码、token 贴进对话或提交进代码；这类信息绝不进 Git。

## 二、上手陌生开源仓库的标准流程

下面是从「看到一个想参与的开源项目」到「提交 PR」的标准步骤。每步都给了示例命令或 prompt（`# PowerShell` 表示在 Windows PowerShell 里运行，`# Git Bash` 表示在 Git Bash 里运行）。

### 步骤 1：Fork & Clone

先把项目「fork（在你自己账号下复制一份）」，再「clone（下载到本地）」。完整流程见 [07-开源贡献与PR指南.md](07-开源贡献与PR指南.md)，这里给最常用的命令：

```powershell
# PowerShell
git clone https://github.com/你的用户名/项目名.git
cd 项目名
```

预期结果：当前目录下出现一个项目文件夹，`cd` 进去后能看到该项目的源代码文件。

### 步骤 2：在项目目录运行 `claude`，先 `/init`

在项目根目录启动 Claude Code，并生成 / 更新项目说明文件 `CLAUDE.md`：

```powershell
# PowerShell
claude
```

启动后，在对话里输入：

```
/init
```

预期结果：Claude Code 会浏览项目，生成或更新一个 `CLAUDE.md` 文件，里面记录了项目的结构、常用命令等。这能让它在后续对话中更懂这个项目。（`/init` 详见 [04-常用Skill与扩展.md](04-常用Skill与扩展.md)。）

### 步骤 3：让 CC 概览架构

用一句 prompt 让它给你讲清楚项目长什么样：

```
请概览这个仓库的目录结构、技术栈和入口文件（程序从哪里开始运行），用简洁的中文说明。
```

预期结果：Claude 会告诉你主要目录各自的职责、用了哪些语言 / 框架、入口在哪。这一步是你「先理解」的关键。

### 步骤 4：找一个适合新手的 issue

在项目的 GitHub 页面，找带 **`good first issue`**（适合新手的第一个任务）标签的 issue。选定后，让 CC 帮你定位相关代码：

```
我想解决这个 issue：<把 issue 的标题和描述粘贴进来>。
请帮我找出与之相关的代码文件和函数，并解释当前的逻辑是怎样的。
```

预期结果：Claude 会指出哪些文件 / 函数与这个问题有关，并解释现状，帮你缩小范围。

### 步骤 5：在新分支上小步修改

**不要直接在主分支上改**。先建一个新分支（分支 = 一条独立的修改线，互不影响）：

```powershell
# PowerShell
git checkout -b fix/简短描述
```

然后让 CC 小步实现改动。建议先让它说方案、你确认后再写代码：

```
请先说明你打算怎么修改（不要直接动手），列出要改哪些文件、改成什么样。
```

确认方案没问题后，再让它实际修改。分支与提交的更多细节见 [06-Git指南.md](06-Git指南.md)。

### 步骤 6：跑测试 / 构建

让 CC 找到并运行这个项目的测试或构建命令：

```
这个项目怎么运行测试和构建？请找到对应命令并运行，把结果告诉我。
```

预期结果：Claude 会找到类似 `npm test`、`pytest`、`make` 之类的命令并执行，反馈是否通过。如果失败，让它一起排查原因。

### 步骤 7：自查

提交前做自查，常用三件套：

- `/code-review`：审查你这次的改动，找 bug 与可简化项（可带 effort 级别，详见 [04-常用Skill与扩展.md](04-常用Skill与扩展.md)）。
- `verify`：真正运行应用，确认改动确实生效。
- 必要时 `security-review`：如果改动涉及鉴权、输入处理、依赖、密钥等敏感点，做一次安全审查。

预期结果：你会得到一份问题清单，逐条处理干净后再进入下一步。

### 步骤 8：提交与开 PR

确认无误后提交并推送，然后在 GitHub 上发起 Pull Request。完整步骤（写好 commit 信息、推送、填 PR 说明）见 [07-开源贡献与PR指南.md](07-开源贡献与PR指南.md)。

## 三、安全与边界

用 AI 协作时，理解它的「默认边界」能让你更放心：

- **CC 默认不会替你提交或推送代码**：`git commit` / `git push` 这类操作通常需要你明确要求，它不会偷偷帮你做。
- **接受前先审阅 diff**：AI 给出的改动，先看一眼「改了哪些行」再接受，别盲目全盘接收。
- **善用 plan mode（计划模式）**：先让 AI 只给方案、不动代码，你确认方案后再执行。这正是「先理解、再改动」的体现。
- **密钥永不入库**：再次强调，API key、密码、token 不要写进代码或提交进 Git。

## 四、端到端示例：给某 Python 小项目修一个小 bug

下面用一个完整叙事，把上面的步骤串起来。假设你在一个 Python 小工具项目里发现了一个 bug：某个函数在输入为空字符串时会崩溃。

**1. Fork 并克隆到本地**

```powershell
# PowerShell
git clone https://github.com/你的用户名/tiny-tool.git
cd tiny-tool
```

**2. 启动 CC 并初始化**

```powershell
# PowerShell
claude
```

在对话里：

```
/init
```

**3. 让它概览项目**

```
请概览这个仓库的目录结构、技术栈和入口文件，用简洁的中文说明。
```

Claude 回复：这是一个用 Python 写的命令行小工具，入口是 `main.py`，核心逻辑在 `tiny_tool/core.py`，测试在 `tests/` 目录，用 `pytest` 运行。

**4. 定位 bug 相关代码**

```
有个 bug：当传入空字符串时程序会崩溃报错。请帮我找到处理输入的函数，
并解释为什么空字符串会导致崩溃。
```

Claude 指出 `core.py` 里的 `parse_input()` 函数没有处理空字符串的情况。

**5. 新建分支并让它给方案**

```powershell
# PowerShell
git checkout -b fix/empty-input-crash
```

```
请先说明修复方案（先别改代码）：你打算怎么处理空字符串？会改哪个函数？
```

Claude 提议：在 `parse_input()` 开头加一个判断，空字符串时返回一个空结果而不是继续处理。你觉得合理，让它动手。

**6. 运行测试**

```
请运行测试，确认现有测试都通过；如果有针对这个 bug 的测试缺失，帮我补一个。
```

Claude 运行 `pytest`，补充了一个「输入空字符串应返回空结果」的测试用例，全部通过。

**7. 自查**

```
/code-review
```

接着让它 `verify`：

```
请运行一下这个工具，用空字符串当输入，确认不再崩溃。
```

预期结果：审查没有遗留问题，实际运行也不再报错。

**8. 提交并开 PR**

按 [06-Git指南.md](06-Git指南.md) 提交，按 [07-开源贡献与PR指南.md](07-开源贡献与PR指南.md) 推送并发起 Pull Request，在 PR 说明里写清楚：修复了什么 bug、怎么修的、加了哪个测试。

到这里，你就完成了一次从克隆到 PR 的完整开源贡献。
