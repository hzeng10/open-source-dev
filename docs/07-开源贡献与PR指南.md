# 07 · 开源贡献与 PR 指南

本指南面向 **Git 与开源贡献的初学者**，教你如何把自己的修改贡献到一个开源项目里。核心载体是 **Pull Request**（简称 PR，中文常说「拉取请求」）——它是你向项目维护者「请求把我的改动合并进去」的标准方式。

> 前置阅读：请先掌握 [06-Git指南.md](06-Git指南.md) 中的分支、提交、远程操作。

---

## 1. 开源协作全景：三个仓库

贡献开源时，你需要同时和 **三份仓库** 打交道。理解它们的关系是不踩坑的前提。

| 名称 | 是谁 | 在哪 |
| --- | --- | --- |
| **upstream**（上游） | 原项目的官方仓库，你**没有**写权限 | GitHub 上别人的仓库 |
| **origin**（你的 fork） | 你点了 Fork 后得到的副本，你**有**写权限 | GitHub 上你的账号下 |
| **本地** | 你电脑上 clone 下来的工作副本 | 你的电脑 |

```
        ┌─────────────────────────┐
        │   upstream（上游原仓库）  │   ← 你只能读，不能直接推
        │  github.com/原作者/项目   │
        └────────────┬────────────┘
              ① Fork │           ▲
                     ▼           │ ⑦ 提 Pull Request
        ┌─────────────────────────┐
        │   origin（你的 fork）     │   ← 你能读能写
        │  github.com/你/项目       │
        └────────────┬────────────┘
         ② clone     │           ▲
                     ▼           │ ⑥ git push
        ┌─────────────────────────┐
        │        本地仓库           │   ← 你在这里写代码
        │      你的电脑             │
        └─────────────────────────┘

        ③ 同步上游：git fetch upstream → 合并 upstream/main 到本地
```

要点：你的改动流向是 **本地 → origin → （PR）→ upstream**；而别人的最新进展流向是 **upstream → 本地**（通过 `fetch upstream`）。

## 2. 标准贡献流程

下面是从零到一份 PR 的完整步骤。

### 步骤 1：在 GitHub 上 Fork

打开目标项目的 GitHub 页面，点右上角的 **Fork** 按钮。稍等片刻，你的账号下就会出现一份同名副本，这就是你的 `origin`。

### 步骤 2：clone 你的 fork，并添加 upstream

```bash
# 终端
git clone https://github.com/你的名字/项目名.git
cd 项目名
git remote add upstream https://github.com/原作者/项目名.git
```

- `clone` 来自 **你的 fork**，所以 `origin` 自动指向你的副本。
- `git remote add upstream <原仓库URL>` 手动添加上游引用，用于日后同步。

验证：

```bash
# 终端
git remote -v
```

预期输出：

```
origin    https://github.com/你的名字/项目名.git (fetch)
origin    https://github.com/你的名字/项目名.git (push)
upstream  https://github.com/原作者/项目名.git (fetch)
upstream  https://github.com/原作者/项目名.git (push)
```

### 步骤 3：同步上游最新代码

在开始改之前，先确保你的 main 分支与上游一致，避免基于过时代码开发。

```bash
# 终端
git fetch upstream            # 下载上游的最新提交
git switch main               # 切到 main 分支
git merge upstream/main       # 把上游 main 合并到本地 main
```

> 也可以用 `git rebase upstream/main` 代替 `merge`。区别：`merge` 会保留一个合并记录，历史更真实；`rebase` 会把你的提交「搬到」上游最新提交之后，历史更线性、更整洁。初学者用 `merge` 更安全；项目若要求线性历史，再学 `rebase`。

### 步骤 4：新建特性分支

**永远不要直接在 main 上改动**。为每个改动新建一个语义化命名的分支。

```bash
# 终端
git switch -c fix/login-crash        # 修复类用 fix/
# 或
git switch -c feat/dark-mode         # 新功能用 feat/
```

### 步骤 5：修改并提交

改完代码后，按 [06-Git指南.md](06-Git指南.md) 的 Conventional Commits 规范提交：

```bash
# 终端
git add .
git commit -m "fix: 修复未登录时点击导致的崩溃"
```

### 步骤 6：推送到你的 fork

```bash
# 终端
git push -u origin fix/login-crash
```

`-u` 会建立分支跟踪关系，之后再推送只需 `git push`。

### 步骤 7：开 Pull Request

有两种方式。

**方式 A：网页方式（推荐新手）**

push 之后，打开你的 fork 页面，GitHub 通常会自动弹出一个 **Compare & pull request** 按钮。点它，确认 base 仓库是 upstream、base 分支是 `main`，填好标题和描述，点 **Create pull request**。

**方式 B：使用 `gh` CLI（命令行）**

`gh` 是 GitHub 官方命令行工具，可在终端直接完成认证、开 PR、查状态等操作。

```bash
# 终端
gh auth login                # 首次使用：登录 GitHub（按提示选择 HTTPS / 浏览器认证）
gh pr create                 # 在当前分支上交互式创建 PR
gh pr status                 # 查看与你相关的 PR 状态
```

`gh pr create` 会提示你填写标题、正文，并确认目标仓库与分支。

`gh pr status` 的预期输出示例：

```
Current branch
  #42  fix: 修复未登录崩溃 [fix/login-crash]

Created by you
  #42  fix: 修复未登录崩溃 [fix/login-crash] - Review required
```

## 3. 提交前检查清单

在开 PR 之前，逐项核对，能大幅提高被接受的概率：

- [ ] **跑通测试和 lint**：本地执行项目的测试与代码检查命令（如 `npm test`、`npm run lint`），确保全绿。具体见 [05-开源项目开发指南.md](05-开源项目开发指南.md)。
- [ ] **阅读并遵守 `CONTRIBUTING.md`**：大多数项目根目录有这份文件，规定了分支命名、提交格式、测试要求等。
- [ ] **提交信息符合规范**：Conventional Commits（见 [06-Git指南.md](06-Git指南.md)）。
- [ ] **关联 issue**：若你的 PR 解决了某个 issue，在描述里写 `Closes #123`，合并后会自动关闭该 issue。
- [ ] **了解 DCO / CLA 要求**：
  - **DCO**（Developer Certificate of Origin，开发者原创声明）：要求每个提交带 `Signed-off-by` 行。用 `-s` 自动添加：
    ```bash
    # 终端
    git commit -s -m "fix: 修复未登录崩溃"
    ```
    提交信息末尾会自动追加：
    ```
    Signed-off-by: 你的名字 <honglizeng1013@gmail.com>
    ```
  - **CLA**（Contributor License Agreement，贡献者许可协议）：部分项目要求你（通常通过机器人在 PR 下）签署一份授权协议后才能合并。按 PR 中机器人的提示操作即可。

## 4. 写好 PR 描述

清晰的描述能让维护者快速理解你的改动。推荐模板：

```markdown
## 背景 / 动机
说明为什么需要这个改动，解决了什么问题。
（如关联 issue：Closes #123）

## 改动内容
- 简要列出主要改了哪些地方
- 每条一行，便于阅读

## 测试
说明你如何验证改动有效（跑了哪些测试、手动测了哪些场景）。

## 截图（如适用）
界面相关改动请附上前后对比图。
```

> 原则：让维护者「不看代码也能看懂你做了什么、为什么、怎么验证的」。

## 5. 回应 Code Review

提交 PR 后，维护者通常会留下评审意见（Code Review）。这是开源协作的常态，请把它当成学习机会而非批评。

### 5.1 根据评审修改并追加提交

按评论修改代码后，正常提交并推送即可，PR 会**自动更新**：

```bash
# 终端
git add .
git commit -m "refactor: 按评审意见抽取公共函数"
git push           # 已建立跟踪关系，直接 push 即可
```

### 5.2 整理提交历史（rebase）

如果评审过程中产生了很多零碎的「修修补补」提交，维护者可能希望你把它们合并整理。可用交互式 rebase（注意：本质上是改写历史）：

```bash
# 终端
git rebase -i HEAD~3        # 整理最近 3 个提交（合并、改信息等）
git push --force-with-lease # 改写历史后需强推；用 --force-with-lease 更安全
```

> `--force-with-lease` 比 `--force` 安全：仅当远程分支没有被别人更新过时才强推，避免覆盖他人改动。只对**你自己的特性分支**这样做。

### 5.3 保持分支与上游同步、解决冲突

评审期间上游可能有了新提交，导致你的 PR 出现冲突。同步上游并解决冲突：

```bash
# 终端
git fetch upstream
git rebase upstream/main      # 或 git merge upstream/main
# 若有冲突：手动编辑冲突文件，然后
git add 冲突文件
git rebase --continue         # 若用的是 merge，则改为 git commit
git push --force-with-lease   # 若做了 rebase
```

冲突文件中会出现这样的标记，保留你想要的内容、删掉标记即可：

```
<<<<<<< HEAD
你的改动
=======
上游的改动
>>>>>>> upstream/main
```

> 解决冲突的思路梳理可以让 Claude Code 协助（见下一节），但**最终保留哪段代码由你决定**。

## 6. 用 Claude Code 辅助贡献

Claude Code（CC）能在贡献流程中帮你不少忙，同时严守安全边界。

- **生成 PR 描述**：「根据我这个分支相对 main 的改动，按背景/改动/测试模板写一份 PR 描述。」
- **自查 diff**：「帮我审一遍这次改动，有没有遗漏、风险或不符合项目规范的地方。」
- **运行评审技能**：CC 提供 `/code-review`（审查代码改动的正确性与可简化点）与 `security-review`（安全审查）等能力，提交前先自查一遍。
- **梳理冲突解决思路**：让 CC 解释冲突双方含义、给出合并建议。

> **安全边界（务必牢记）**：提交（commit）、推送（push）、开 PR 等**外发操作需要你明确确认，CC 不会擅自执行**。它会把命令准备好、解释清楚，由你拍板。这能防止「还没想好就把半成品推到了公开仓库」。

更多 CC 与 IDE 的集成见 [08-IDE集成指南.md](08-IDE集成指南.md)。

## 7. 开源礼仪

技术之外，社区氛围同样重要：

- **保持 PR 小而专注**：一个 PR 只做一件事。大而杂的 PR 难以评审，容易被搁置。
- **尊重维护者的时间**：他们大多是志愿者。描述清楚、自测充分，能减轻他们负担。
- **耐心等待**：评审可能需要几天甚至几周。礼貌地催一次没问题，反复催促则不妥。
- **遵守行为准则（Code of Conduct）**：许多项目有 `CODE_OF_CONDUCT.md`，保持友善、专业、包容。
- **被拒绝也没关系**：不是所有 PR 都会被合并。理解维护者的取舍，从反馈中学习。
