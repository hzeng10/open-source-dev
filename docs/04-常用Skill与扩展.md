# 04 - 常用 Skill 与扩展

> 返回 [📖 文档索引](README.md)

本篇介绍 Claude Code 的「Skill（技能）」以及其他几种扩展能力（subagent 子代理、MCP）。读完你会知道：常用内置 Skill 都能做什么、如何触发它们，以及如何自己写一个最小的自定义 Skill。

阅读本篇前，建议先看过 [03-配置文件详解.md](03-配置文件详解.md)，了解配置文件大致放在哪里。

---

## 一、什么是 Skill

**Skill（技能）** 是 Claude Code 提供的「可复用能力包」：把一套固定的工作流程或专门指令打包成一个名字，你只要用 **斜杠命令**（slash command，以 `/` 开头输入的快捷命令）`/<skill 名>` 就能触发它，而不必每次都重新描述一长串要求。

举个例子：与其每次手动输入「请帮我审查刚才改动的代码，找出 bug 和可以简化的地方」，你直接输入 `/code-review` 就行——Skill 内部已经写好了这套流程。

它和几个相近概念的关系（各一句话）：

- **slash command（斜杠命令）**：是触发 Skill 的「按钮」；很多 Skill 就是以斜杠命令的形式暴露给你的（如 `/init`、`/code-review`）。
- **subagent（子代理）**：是主 AI 临时派出去、专门干某个子任务的「分身」；有些 Skill 在内部会调用一个或多个子代理来并行干活。
- **MCP**：是一套让 Claude Code 连接「外部工具和数据源」（如文件系统、数据库）的开放协议；Skill 关注「做什么流程」，MCP 关注「能连到哪些外部能力」。

> 💡 小贴士：随时输入 `/help` 可以查看当前版本可用的斜杠命令与 Skill 列表。

## 二、常用内置 Skill 介绍

下表是几个常见的内置 Skill。**注意：不同版本可用的 Skill 不同**，请始终以你本地 `/help` 的输出或工具列表为准，本表仅作入门参考。

| Skill / 命令 | 用途 | 何时用 | 如何触发 |
| --- | --- | --- | --- |
| `/init` | 为当前项目生成 / 更新 `CLAUDE.md` 说明文件 | 刚进入一个新项目、想让 Claude Code 了解项目约定时 | 在项目目录下输入 `/init` |
| `/code-review` | 审查当前 diff（你这次的代码改动），找出 bug 与可简化项 | 改完代码、提交前自查时 | 输入 `/code-review`，可带 effort 级别 |
| `verify` | 真正运行应用，观察改动是否生效 | 想确认「这个修复 / 功能确实能跑通」时 | 让 Claude「验证一下这个改动」即可触发 |
| `security-review` | 对当前分支的改动做安全审查 | 改动涉及鉴权、输入处理、依赖、密钥等敏感点时 | 让 Claude 做「安全审查」，或用对应命令触发 |
| `deep-research` | 多来源联网深度研究，产出带引用的报告 | 需要查资料、对比方案、了解某个陌生技术时 | 让 Claude 做「深度研究」并给出问题 |
| `simplify` | 对刚才的改动做精简重构（只做质量优化，不找 bug） | 代码能跑但写得啰嗦、想让它更干净时 | 让 Claude「精简一下这些改动」 |
| `loop` | 按设定的间隔，重复运行某个 prompt 或命令 | 需要定时轮询状态、反复执行同一任务时 | 例如 `/loop 5m /code-review`（每 5 分钟跑一次） |
| `schedule` | 创建按 cron 规则在云端定时运行的代理 | 想让某个任务「每天定点自动跑」时 | 让 Claude「创建一个定时任务」并描述时间规则 |

### 关于 `/code-review` 的 effort 级别

`/code-review` 支持不同的「努力级别（effort）」，决定审查的深浅与覆盖面：

- **low / medium**：更少但更有把握的发现，速度快，适合日常小改动。
- **high → max**：覆盖更广，可能包含一些不确定的发现，适合重要改动。
- **`/code-review ultra`**：在云端用多个子代理做深度审查，最彻底，适合大型或关键改动。

预期结果：运行后，Claude 会列出它发现的问题（bug、可简化点等），通常按严重程度或文件分组，并给出修改建议。

> 💡 进一步的代码审查与排错技巧见 [09-最佳实践与排错.md](09-最佳实践与排错.md)。

## 三、Skill 的安装与部署

### 1. 内置 Skill：直接用

内置 Skill 无需安装，直接用 `/名字` 调用即可，例如 `/init`、`/code-review`。不确定有哪些时，先 `/help`。

### 2. 用户级 vs 项目级 Skill 目录

你也可以放置自定义 Skill，按作用范围分两种：

- **用户级（个人全局）**：通常放在用户主目录下的 `~/.claude/skills/`，对你所有项目都生效。
  - 在 Windows PowerShell 中，`~` 一般对应 `C:\Users\你的用户名`，即 `C:\Users\你的用户名\.claude\skills\`。
- **项目级（随项目走）**：放在项目根目录的 `.claude/skills/`，只对该项目生效，并可随项目一起提交给团队共享。

> ⚠️ 目录位置与命名可能随版本变化，请以官方文档为准：<https://docs.anthropic.com/en/docs/claude-code>

### 3. 自定义一个最小 Skill

一个最小的 Skill 就是一个名为 `SKILL.md` 的文件，放在「用户级或项目级 skills 目录下、以 Skill 名命名的子文件夹」里。结构如下：

```
.claude/
  skills/
    hello-greeting/      # 文件夹名即 Skill 名
      SKILL.md
```

`SKILL.md` 内容模板（顶部 `---` 之间的部分叫 **frontmatter**，是给工具读的元信息；下方正文是真正的指令）：

```markdown
---
name: hello-greeting
description: 用友好的中文向用户打招呼，并简单介绍当前项目。
---

当被触发时，请执行以下步骤：

1. 用一句友好的中文向用户问好。
2. 查看当前目录，用一两句话概括这个项目是做什么的。
3. 提醒用户可以输入 /help 查看更多可用命令。
```

要点说明：

- **`name`**：Skill 的名字，建议与文件夹名一致，最好用小写字母和连字符。
- **`description`**：一句话说明这个 Skill 干什么。Claude Code 会根据它来判断「什么时候该用这个 Skill」，写清楚很重要。
- **正文**：就是你希望 Claude 每次触发时执行的指令。

放好文件后，重新启动 `claude`（或按官方说明刷新），Claude Code 就能发现这个 Skill。之后你描述相关需求时它可能自动调用，或者你也可以显式触发它。

预期结果：触发后，Claude 会按 `SKILL.md` 正文里写的步骤来执行，而不需要你再重复一遍这些要求。

## 四、subagent 子代理简介

**subagent（子代理）** 是主 AI 派生出来、专门处理某个子任务的独立 AI 实例。它有自己独立的「上下文窗口」（可以理解为独立的工作记忆），干完活只把**结论**带回给主 AI，而不会把一路读过的大量文件内容都塞进主对话里。

它的两个主要好处：

- **节省主对话的「记忆空间」**：把繁重的搜索、阅读交给子代理，主对话保持清爽。
- **支持并行**：可以同时派出多个子代理分别处理互不依赖的任务，更快出结果。

对初学者来说，你通常不需要手动管理子代理——当你交给 Claude Code 一个需要「翻很多文件」的大任务时，它可能会自动用子代理去完成。了解这个概念，能帮你理解它为什么有时会说「正在派一个代理去搜索……」。

## 五、MCP 扩展简介

**MCP（Model Context Protocol，模型上下文协议）** 是一个开放标准，让 Claude Code 能连接「外部工具和数据源」。装好一个 **MCP server（MCP 服务）** 后，Claude 就多了一组新能力，比如：

- **文件系统类**：让 Claude 以受控方式读写某个指定目录。
- **数据库类**：让 Claude 直接查询某个数据库（如 PostgreSQL）。

你一般通过在 settings 配置文件里增加一个 MCP server 条目来启用它，概念上类似这样（仅示意，具体字段以官方为准）：

```jsonc
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "D:/dev/some-project"]
    }
  }
}
```

上面这段的含义：注册一个名为 `filesystem` 的 MCP server，启动方式是运行 `npx ...` 这条命令，并把 `D:/dev/some-project` 作为它可访问的目录。

> ⚠️ MCP 的字段、配置文件位置可能随版本变化，配置文件放在哪、怎么写，详见 [03-配置文件详解.md](03-配置文件详解.md)，并以官方文档为准。
